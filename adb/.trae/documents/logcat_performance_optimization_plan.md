# Logcat 性能优化方案

## 问题分析

通过代码审查，发现以下导致卡顿和未响应的关键问题：

### 问题 1：主线程阻塞 - 设备检查
**位置**: `main_window.py:_start_logcat()` (L350-389)
- 使用 `QApplication.processEvents()` 和 `setCursor(Qt.WaitCursor)` 在主线程同步检查设备
- 当 ADB 响应慢时，UI 会冻结

### 问题 2：主线程处理日志过滤
**位置**: `main_window.py:_on_log_batch_received()` (L405-424)
- 每批日志（50条）都在主线程进行过滤匹配
- 当日志量大时，主线程忙于处理数据，无法响应 UI 事件

### 问题 3：逐条插入 QTextEdit
**位置**: `main_window.py:_append_log_batch()` (L426-441)
- 使用 `QTextCursor.insertText()` 逐条插入日志
- 每条日志都触发一次 UI 更新，50条 = 50次渲染

### 问题 4：全量重建日志显示
**位置**: `main_window.py:_apply_filter()` (L454-475)
- 每次过滤条件变化时，`clear()` 后逐条重新插入所有日志
- 如果有 10000 条日志，会执行 10000 次插入操作

### 问题 5：日志存储无上限控制
**位置**: `main_window.py:_on_log_batch_received()` (L409)
- 虽然限制了 10000 条，但裁剪操作 `self._log_entries[-5000:]` 会创建新列表
- 频繁的大列表操作消耗内存和 CPU

## 优化方案

### 优化 1：异步设备检查（优先级：高）
**方案**: 将设备检查移到后台线程
```python
# 使用 QThread 或 QTimer 异步检查设备
# 避免在 _start_logcat 中同步等待
```

### 优化 2：日志预格式化 + 批量插入（优先级：高）
**方案**: 
- 在 LogcatService 中预先格式化日志为 HTML
- 使用 `QTextEdit.append()` 或 `setHtml()` 批量更新
- 减少 UI 更新频率，使用定时器合并批次

### 优化 3：后台过滤处理（优先级：高）
**方案**:
- 将过滤逻辑移到工作线程
- 主线程只负责显示已过滤的结果
- 使用信号槽传递过滤后的数据

### 优化 4：优化过滤刷新（优先级：中）
**方案**:
- 不重建整个日志视图
- 使用 `QTextDocument` 的搜索功能高亮匹配项
- 或者使用虚拟列表（只渲染可见区域）

### 优化 5：增加日志批次间隔（优先级：中）
**方案**:
- 增大 `_batch_size` 从 50 到 100-200
- 增大 `_batch_timeout` 从 0.1s 到 0.2-0.5s
- 减少信号发射频率

### 优化 6：使用更高效的日志显示组件（优先级：低，长期方案）
**方案**:
- 替换 `QTextEdit` 为自定义 `QAbstractItemView` + `QListView`
- 实现虚拟滚动，只渲染可见区域的日志
- 适合超大量日志场景（10万+条）

## 推荐实施顺序

### 第一阶段：快速优化（改动小，效果明显）
1. 增大批次大小和超时时间
2. 批量插入日志（合并批次，减少 UI 更新次数）
3. 预格式化日志文本，减少主线程计算

### 第二阶段：架构优化（改动中等，效果显著）
4. 异步设备检查
5. 后台过滤处理
6. 优化过滤刷新逻辑

### 第三阶段：长期优化（改动大，按需实施）
7. 替换为虚拟列表组件
